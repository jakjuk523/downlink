# Downlink

Универсальный загрузчик видео с YouTube, VK Видео и RuTube.
Тёмный интерфейс, большие кнопки, живой прогресс, устанавливается как PWA.

## Как это устроено (важно понимать перед деплоем на GitHub Pages)

**GitHub Pages отдаёт только статику** — HTML/CSS/JS. Он не умеет запускать
Python и yt-dlp, а именно yt-dlp умеет доставать реальные потоки с
YouTube/VK/RuTube и склеивать видео+звук. Поэтому в проекте два отдельных
слоя, и это не изменится, на каком бы хостинге ни лежал сайт:

```
frontend/   — статический сайт и PWA. Это то, что грузится на GitHub Pages
              (или открывается локально). Экран, кнопки, прогресс-бары.
backend/    — Python-сервер (Flask + yt-dlp). Именно он качает видео.
              GitHub Pages его не запустит — нужен обычный Python-хостинг.
```

Если просто закинуть всю папку на GitHub и включить Pages — сайт откроется
и будет красивым, но кнопка «Загрузить» не будет работать, пока где-то не
запущен `backend/app.py`.

### Два сценария использования

**1. Только для себя, локально (как раньше)**
Ничего решать не нужно: `Запустить.bat` / `run.sh` поднимают backend, а он
сам же отдаёт и frontend на `http://127.0.0.1:8642`. Всё работает из коробки.

**2. Как публичный сайт (то, что ты описал)**
- `frontend/` — грузишь в репозиторий, включаешь GitHub Pages (Settings →
  Pages → Branch: main → root). Получаешь ссылку вида
  `https://<юзернейм>.github.io/<репозиторий>/`.
- `backend/` — грузишь на любой хостинг, который умеет запускать Python
  24/7 и слушать входящие соединения: Render, Railway, Fly.io, обычный VPS
  и т.п. Названия и текущие условия/тарифы этих сервисов меняются, так что
  перед выбором стоит свериться с их актуальными сайтами.
- В `frontend/app.js` в самом верху есть строка:
  ```js
  const DEFAULT_API_BASE = ""; // например: "https://downlink-backend.onrender.com"
  ```
  Вписываешь туда адрес задеплоенного backend — и GitHub Pages сайт будет
  ходить за загрузками именно туда. (Это же можно временно поменять прямо
  в браузере: Shift + клик по индикатору-лампочке вверху страницы.)
- В `backend/app.py` для продакшена стоит сузить CORS с `"*"` до конкретного
  адреса твоего GitHub Pages — строчка с комментарием рядом уже подготовлена.

Файл, который скачал посетитель сайта, теперь не остаётся лежать на
сервере — после завершения загрузки на карточке видео появляется кнопка
**«💾 Сохранить файл»**, она отдаёт готовый файл через браузер, как обычная
ссылка на скачивание (это работает и локально, и через удалённый backend).

## Формат PWA

- `manifest.webmanifest` — имя, иконки, тема оформления, `display: standalone`.
- `sw.js` — кэширует только саму оболочку сайта (HTML/JS/иконки), все
  запросы к `/api/*` всегда идут в сеть напрямую — прогресс и файлы никогда
  не читаются из кэша.
- Кнопка «⬇ Установить как приложение» появляется автоматически, когда
  браузер считает сайт готовым к установке (нужен HTTPS — GitHub Pages это
  обеспечивает сам).
- Иконки (`frontend/icons/`) и превью для соцсетей (`frontend/preview.png`)
  уже сгенерированы и лежат в проекте.

## Установка ffmpeg (нужен для сборки видео+звук в один файл)

- Windows: https://www.gyan.dev/ffmpeg/builds/ — добавить папку `bin` в PATH.
- macOS: `brew install ffmpeg`
- Linux: `sudo apt install ffmpeg`

На облачном хостинге ffmpeg нужно поставить как системный пакет — на
большинстве Python-хостингов это делается через `apt.txt` / build-скрипт /
Dockerfile, конкретный способ зависит от выбранной платформы.

## Форматы скачивания

240p · 360p · 480p · 720p · 1080p · 2K (1440p) · 4K (2160p) · MP3 (только звук).

## Продажа как готового продукта

- Один и тот же `frontend/` без единой правки одинаково открывается и
  локально, и с GitHub Pages, и с любого другого статического хостинга —
  меняется только `DEFAULT_API_BASE`.
- Перед публикацией на маркетплейсе стоит явно указать в описании, что
  инструмент предназначен для личного контента и общедоступных роликов —
  ответственность за соблюдение правил конкретной площадки (YouTube, VK,
  RuTube) и авторских прав лежит на пользователе сайта.
